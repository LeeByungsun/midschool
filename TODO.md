## 현재 추천 우선순위

### 1순위 — 접근성/반응형 보강
- 이유: 공통 상태 컴포넌트가 생긴 만큼 키보드 흐름과 다양한 화면 폭에서의 일관성 점검 효과가 커짐
- 완료 기준: 키보드 탐색/포커스/터치 타깃 점검 + 태블릿/데스크톱 레이아웃 조정

### 2순위 — PWA/알림 검토
- 이유: 조회 기능의 기본 틀이 갖춰져서 이제 설치성/빠른 접근/브라우저 알림 같은 웹 고유 개선을 검토할 시점임
- 완료 기준: manifest/설치 흐름/알림 fallback 방향 정리

### 3순위 — 캐시 고도화
- 이유: 기본 캐시/복구 전략은 적용됐고, 다음 단계에서는 오프라인 대응과 캐시 표시 UX 세분화가 필요함
- 완료 기준: 오프라인 fallback 범위 정리 + 캐시 안내/만료 표시 정교화

### 최후순위 — 타이머 실제 동작
- 이유: 현재 웹의 핵심 조회 흐름(설정/시간표/급식/일정) 안정화가 타이머보다 우선이며, 사용자 요청에 따라 가장 뒤로 미룸
- 완료 기준: 시작/일시정지/재시작/리셋 + 새로고침 복원 + 완료 UX

---

## 완료 기준 메모

- 설정 저장 후 새로고침해도 학년/반이 유지되어야 함
- 브라우저는 직접 NEIS API를 때리지 않고 `/api/*` 경유 구조여야 함
- 대시보드는 목업이 아니라 실제 데이터를 우선 사용해야 함
- 실데이터 연결 후 로딩/빈 상태/오류 상태가 모두 존재해야 함
- 타이머는 최소한 새로고침 복원 전까지 구현되어야 “기본 기능 완료”로 봄
