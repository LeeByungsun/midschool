# AGENTS Guidelines for Web

이 문서는 `web/` 하위에 적용되는 웹 전용 작업 규칙입니다.  
루트 `AGENTS.md`의 상위 규칙을 따르되, 웹 코드에는 아래 기준을 우선 적용합니다.

## 1. 기본 방향
- `web/`은 `중학교도우미`의 웹 클라이언트 영역입니다.
- 제품 요구사항의 기준 문서는 `docs/project_specification.md` 입니다.
- 공통 동작은 해당 문서의 **공통 요구사항**을 우선 따릅니다.
- 웹에서만 달라지는 부분은 **Web 전용 명세**를 따릅니다.
- 특별한 합의가 없으면 웹은 **Next.js App Router 기반 단일 웹앱**으로 유지합니다.

## 2. 프레임워크/라우팅 규칙
- 웹은 **Next.js + TypeScript + App Router** 기준을 우선합니다.
- 특별한 이유가 없으면 **Pages Router를 도입하지 않습니다.**
- 라우팅은 `app/` 디렉터리 기준으로 구성합니다.
- 페이지는 `app/**/page.tsx`, 레이아웃은 `app/**/layout.tsx` 규칙을 따릅니다.
- API 경계는 `app/api/**/route.ts` 로 둡니다.
- 프레임워크를 바꾸거나 `src/` 구조, monorepo 구조 등 큰 변경을 하려면 문서와 구조 기준을 먼저 갱신합니다.

## 3. 서버/클라이언트 컴포넌트 규칙
- **기본은 Server Component** 입니다.
- `use client`는 아래 경우에만 추가합니다.
  - 브라우저 이벤트 처리
  - 상태/이펙트 훅 사용
  - 브라우저 전용 API 접근
  - 클라이언트 전용 라이브러리 사용
- 단순 데이터 표시, 초기 데이터 로딩, 레이아웃 구성은 가능하면 서버 컴포넌트로 둡니다.
- 서버 전용 로직을 클라이언트 컴포넌트로 밀어 넣지 않습니다.
- 클라이언트 컴포넌트는 가능한 한 얇게 유지합니다.

## 4. 데이터/연동 규칙
- NEIS API 같은 외부 연동은 가능하면 `app/api/**/route.ts` 또는 서버 전용 유틸을 통해 감쌉니다.
- **브라우저에서 NEIS API를 직접 호출하지 않습니다.**
- API 키, 비밀값, 서버 전용 설정은 클라이언트 코드에 노출하지 않습니다.
- `NEXT_PUBLIC_*` 환경변수는 브라우저 노출이 허용된 값에만 사용합니다.
- 시간표/급식/학사 일정 응답 매핑은 `lib/` 같은 공용 계층으로 분리합니다.
- 공통 에러 문구와 응답 해석 규칙은 UI 컴포넌트 안에 흩뿌리지 말고, 재사용 가능한 계층으로 모읍니다.
- 외부 API 응답 원형을 바로 화면에 바인딩하지 말고, 웹 UI에 맞는 타입으로 한 번 정리합니다.

## 5. 상태/저장 규칙
- 서버 데이터와 로컬 UI 상태를 구분합니다.
- 간단한 사용자 설정(학년/반, 최근 선택값 등)은 웹 저장소를 사용할 수 있습니다.
- `localStorage` 접근은 직접 흩어 쓰지 말고 래퍼 유틸/모듈을 두는 쪽을 우선합니다.
- `window`, `document`, `localStorage`, `Notification` 같은 브라우저 API는 클라이언트 경계 안에서만 접근합니다.
- 타이머 상태는 새로고침/재방문 시 복원 가능성을 고려해 설계합니다.
- 초기에 전역 상태 라이브러리를 무조건 도입하지 말고, React 기본 상태/서버 데이터 구조로 충분한지 먼저 판단합니다.

## 6. UI/UX 규칙
- 기본은 **모바일 퍼스트**입니다.
- 홈 화면은 대시보드형 요약 구조를 우선합니다.
- 웹은 Android UI를 그대로 복붙하지 말고, 웹에 맞는 레이아웃/상호작용으로 번역합니다.
- 접근성(시맨틱 마크업, 포커스 이동, 버튼/폼 라벨)을 기본 품질 기준으로 봅니다.
- 기본 HTML 시맨틱을 우선하고, ARIA는 정말 필요할 때만 보강합니다.
- 텍스트 하드코딩은 작은 초기 스캐폴드에서는 허용될 수 있지만, 기능이 커지면 문자열 관리 전략을 분리합니다.
- 대시보드/상세/설정/타이머는 재사용 가능한 카드/섹션 구조를 우선합니다.

## 7. 컴포넌트/폴더 규칙
- 페이지 라우트: `app/**/page.tsx`
- API 경계: `app/api/**/route.ts`
- 재사용 UI: `components/`
- 도메인/연동 유틸: `lib/`
- 브라우저 저장소/타이머 유틸: `lib/storage/`
- 외부 API 연동 유틸: `lib/neis/`
- 스타일/토큰은 프로젝트 스타일 체계에 맞춰 한곳에서 관리합니다.
- 페이지 파일에 과도한 비즈니스 로직을 넣지 않습니다.
- 컴포넌트는 “페이지 전용”, “재사용 UI”, “도메인 보조” 책임을 구분합니다.

## 8. 품질/검증 규칙
- 구현 후에는 가능한 범위에서 `lint`, `typecheck`, `build` 를 확인합니다.
- 데이터 경계 변경 시에는 성공/에러/빈 데이터 상태를 같이 점검합니다.
- 모바일 폭 기준과 일반 데스크톱 폭 기준 모두에서 레이아웃을 확인합니다.
- 접근성 관련 변경 시에는 키보드 이동과 폼 라벨 상태를 같이 확인합니다.

## 9. 구현 우선순위
- 1차:
  - 설정 저장
  - 대시보드
  - 시간표 조회
  - 학사 일정 조회
- 2차:
  - 타이머
  - 에러 상태 UX 강화
  - 캐시 개선
- 3차:
  - PWA
  - 브라우저 알림
  - 오프라인/설치 UX

## 10. 하지 말아야 할 것
- Android Activity/XML 구조를 웹에 그대로 옮기기
- Pages Router와 App Router를 혼합해서 구조를 흐리기
- 기본이 서버 컴포넌트인데 습관적으로 모든 파일에 `use client` 붙이기
- 브라우저 클라이언트에서 민감한 외부 API 호출 세부사항 노출하기
- 초기 단계부터 과도한 상태관리/복잡한 추상화 도입하기
- 문서 기준 없이 프레임워크/구조를 임의로 바꾸기

## 11. 문서 반영 규칙
- 웹 구조나 정책이 바뀌면 다음 문서를 함께 갱신합니다.
  - `docs/project_specification.md`
  - `docs/project-structure.md`
  - 필요 시 `README.md`
- 웹 자동화 규칙이나 반복 작업 흐름이 생기면 `.codex/skills/` 또는 `.codex/prompts/` 아래에 정리합니다.
