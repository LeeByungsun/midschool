# 학교도우미 멀티플랫폼 프로젝트 기획서

이 문서는 `MisSchoolApp` 저장소 안에서 운영되는 **학교도우미** 서비스의 멀티플랫폼 기획과 현재 구현 상태를 정리한 문서입니다.

이 문서는 기존 단일 Android 중심 기획서를 **공통 요구사항**, **Android 전용 사항**, **Web 전용 사항**으로 나누어 다시 정리한 버전입니다.

---

## 1. 프로젝트 개요 (Overview)

- **서비스 이름**: 학교도우미
- **주요 타겟**: 학교 생활 정보를 자주 확인해야 하는 학생
- **서비스 목표**: 학생들이 매일 확인해야 하는 시간표, 급식, 학사 일정 정보와 자기주도학습용 타이머를 하나의 서비스로 쉽게 이용할 수 있게 돕는 것
- **플랫폼 전략**
  - **Android**: 현재 구현/운영의 기준 플랫폼
  - **Web**: 같은 도메인 기능을 브라우저에서도 사용할 수 있도록 확장 및 구현 중

---

## 2. 공통 요구사항 (Shared Specification)

이 섹션은 Android와 Web 양쪽에 공통으로 적용되는 제품 요구사항입니다.

### 2.1 공통 핵심 기능

#### 1) 사용자 기본 설정
- 사용자는 자신의 **학년/반 정보**를 입력하고 저장할 수 있어야 합니다.
- 웹에서는 학교 검색을 통해 **학교/교육청 코드와 학교 종류(초등학교/중학교)** 를 함께 저장할 수 있어야 합니다.
- 서비스는 이 정보를 기반으로 시간표/급식/학사 일정 조회에 필요한 기본 문맥을 구성합니다.

#### 2) 메인 대시보드
- 사용자가 앱 또는 웹에 진입했을 때 주요 정보를 한눈에 볼 수 있어야 합니다.
- 기본적으로 다음 정보를 요약 제공합니다.
  - 오늘 날짜
  - 오늘 수업(시간표) 요약
  - 오늘 급식 요약
  - 다가오는 학사 일정 요약

#### 3) 시간표 조회
- 사용자의 학년/반 정보를 기준으로 **일간 시간표**를 조회할 수 있어야 합니다.
- 날짜를 이동하며 특정 날짜의 수업 목록을 확인할 수 있어야 합니다.
- 교시별 과목 정보를 읽기 쉽게 나열해야 합니다.

#### 4) 학사 일정 조회
- 해당 월 또는 가까운 기간의 학교 일정을 확인할 수 있어야 합니다.
- 주요 행사 및 일정이 리스트 형태로 제공되어야 합니다.

#### 5) 급식 정보 조회
- 날짜 기준 급식 정보를 조회할 수 있어야 합니다.
- 메인 화면에서는 요약 정보를, 상세 화면/영역에서는 보다 읽기 쉬운 형태의 정보를 제공할 수 있어야 합니다.
- 웹 상세 화면에서는 메뉴 목록, 알레르기 표기, 칼로리, 영양 정보, 원산지 정보를 날짜별로 확인할 수 있어야 합니다.

#### 6) 스터디 타이머
- 집중/휴식 같은 프리셋 기반 타이머를 제공해야 합니다.
- 사용자는 진행 상태를 쉽게 인지할 수 있어야 하며, 앱/웹을 다시 열어도 가능한 범위에서 상태가 유지되어야 합니다.

### 2.2 공통 데이터 정책

- **데이터 소스**: [나이스(NEIS) 교육정보 개방 포털 API](https://open.neis.go.kr/)
- **사용 API**
  - 급식: `hub/mealServiceDietInfo`
  - 학사 일정: `hub/SchoolSchedule`
  - 중학교 시간표: `hub/misTimetable`
  - 초등학교 시간표: `hub/elsTimetable`
- **캐시 전략**
  - 급식 캐시: 12시간
  - 시간표 캐시: 24시간
  - 학사 일정 캐시: 12시간
- **오류 대응 기본 원칙**
  - 네트워크 오류, 응답 없음, 데이터 없음, 사용자 설정 누락을 구분해서 보여줄 것
  - 마지막 성공 데이터가 있으면 가능한 범위에서 재사용할 것
  - 웹에서는 신선한 캐시가 있으면 우선 재사용하고, 최신 요청 실패 시 마지막 성공 데이터를 fallback으로 표시할 수 있어야 함

### 2.3 공통 도메인/아키텍처 원칙

- UI와 데이터 계층은 분리합니다.
- 학생 정보, 학교 데이터, 캐시, 설정은 명확히 분리된 책임으로 관리합니다.
- 플랫폼별 구현은 달라도, 다음 도메인 개념은 공통으로 유지합니다.
  - 학생 정보
  - 시간표 항목
  - 급식 정보
  - 학사 일정
  - 사용자 설정
  - 타이머 상태

## 3. Android 전용 명세 (Current Android Scope)

이 섹션은 **현재 구현되어 있거나 Android 플랫폼에 특화된 요소**입니다.

### 3.1 Android 화면 구성

- **스플래시 화면 (`SplashActivity`)**
  - 앱 진입 시 초기 데이터 세팅 및 권한 체크 담당
- **초기 설정 (`SetupActivity`)**
  - 첫 실행 시 학년/반 입력
- **설정 (`SettingsActivity`)**
  - 학교 정보, 학년/반, 타이머 관련 상세 옵션 관리
- **메인 대시보드 (`MainActivity`)**
  - 주요 정보 요약 카드 및 타이머 표시
- **시간표 화면 (`TimetableActivity`)**
  - 날짜 이동형 일간 시간표 조회
- **학사 일정 화면 (`ScheduleActivity`)**
  - 월간 일정 목록 확인

### 3.2 Android 전용 UX/기능

- **홈 화면 위젯 (`widget/`)**
  - 오늘/내일 시간표를 앱 외부에서 볼 수 있는 AppWidget 제공
  - 위젯 본문 탭 시 앱으로 바로 진입 가능
  - 날짜 변경, 시간 변경, 시간대 변경, 재부팅 시 자동 갱신
  - 위젯 설정 화면에서 `내일 시간표 표시` 여부 제어
  - 작은 위젯 대응 및 과목명 길이 제한 적용

- **Android 알림/백그라운드 동작**
  - `TimerAlarmReceiver` 기반 타이머 완료 알림
  - 로컬 노티피케이션 및 진동 지원

- **Android 전용 시각 구성**
  - 숫자형 타이머와 `TimerRingView` 커스텀 링 뷰 지원
  - Activity + XML 레이아웃 기반 화면 구성

### 3.3 Android 기술 스택

- **언어/런타임**: Kotlin, Android SDK
- **최소/대상 SDK**: minSdk 26, targetSdk 36
- **아키텍처**: MVVM
- **UI 계층**: Activity + DataBinding + ViewModel
- **비동기 처리**: Kotlin Coroutines, Flow, `runCatching`
- **네트워크**: Retrofit2 + OkHttp3 + Gson
- **의존성 주입**: Dagger Hilt
- **로컬 저장소**: `SharedPreferences` 기반 저장 및 간단 캐시

## 4. Web 전용 명세 (Current + Planned Web Scope)

이 섹션은 **웹 확장 시 별도로 고려해야 하는 사항**입니다.  
현재는 일부 기능이 실제 구현된 상태이며, 아래 내용은 **현재 반영된 범위 + 남은 설계 기준**을 함께 설명합니다.

### 4.1 Web 목표

- Android와 동일한 핵심 기능을 브라우저에서도 사용할 수 있어야 합니다.
- 별도 앱 설치 없이 시간표/급식/학사 일정/타이머를 빠르게 확인할 수 있어야 합니다.
- 모바일 브라우저와 데스크톱 브라우저 모두에서 읽기 쉬운 반응형 UI가 필요합니다.

### 4.2 Web에서 우선 고려할 화면/기능

- **홈 대시보드**
  - 오늘 날짜
  - 오늘 시간표 요약
  - 오늘 급식 요약
  - 다가오는 일정 요약
  - 최근 가정통신문 미리보기
  - 타이머 현재 상태 및 오늘 기록 요약
- **시간표 페이지**
  - 날짜 이동형 조회
- **급식 페이지**
  - 날짜 이동형 상세 조회
  - 메뉴/알레르기/칼로리/영양/원산지 표시
- **학사 일정 페이지**
  - 월 단위 또는 리스트형 조회
- **설정 페이지**
  - 학교 검색, 학년/반 등 개인 설정 저장
- **타이머 영역**
  - 집중/휴식 프리셋 제공
- **가정통신문 미리보기**
  - 학교 홈페이지 게시판 패턴을 해석해 최근 가정통신문을 요약 표시
  - 지원하지 않는 학교 홈페이지는 복구 가능한 안내 문구로 처리

### 4.2.1 현재 웹 구현 상태

- Next.js App Router 기반 단일 웹앱 구조를 사용합니다.
- 웹 브랜드 표기는 `학교도우미`를 사용합니다.
- 주요 페이지
  - `/` 홈 대시보드(시간표/급식/일정/가정통신문/타이머 요약)
  - `/setup`, `/settings` 설정/초기 설정
  - `/timetable` 시간표
  - `/schedule` 학사 일정
  - `/meals` 급식 상세
  - `/timer` 타이머(집중/휴식/딥포커스 프리셋, 시작/일시정지/재시작/리셋, 새로고침 복원, 오늘 기록/알림 설정 지원)
- 학교 검색
  - 초등학교/중학교 검색 지원
  - Enter 키로 학교 검색 가능
  - 학교 코드/교육청 코드/학교 종류를 저장해 이후 조회에 재사용
- 공통 상태 UX
  - 로딩 / 오류 / 빈 상태 / 설정 필요 상태를 공용 컴포넌트로 정리
  - 재시도 버튼 패턴을 공통화
- 캐시/복구 전략
  - 급식/시간표/일정 데이터를 브라우저 저장소 캐시에 저장
  - 신선한 캐시가 있으면 우선 표시
  - 최신 요청 실패 시 마지막 성공 데이터를 fallback으로 표시
  - 캐시 사용 여부와 fallback 여부를 화면 문구로 안내
- 가정통신문 미리보기
  - `app/api/notices/route.ts` 경유로 학교 홈페이지에서 최근 가정통신문을 수집
  - 학교 코드 기반 홈페이지 확인 뒤 게시판 URL 패턴을 해석해 최근 항목을 반환
  - 일부 교육청/학교 홈페이지는 SSO 또는 구조 제한으로 미지원일 수 있으며, 이 경우 복구 가능한 안내 메시지를 반환
- 타이머 상태 복구
  - 타이머의 남은 시간과 진행 상태를 브라우저 저장소에 저장
  - 실행 중인 타이머는 목표 종료 시각을 기준으로 새로고침 후에도 복원
  - 완료 시 즉시 완료 상태 UX로 전환
  - 홈 대시보드와 타이머 페이지가 현재 상태와 오늘 기록을 함께 공유
  - 홈 대시보드의 타이머 요약 카드는 별도 컴포넌트로 분리하되 같은 저장소/구독 훅을 사용해 단일 상태원을 유지
  - 브라우저 Notification API 지원 환경에서는 완료 알림을 선택적으로 사용할 수 있음

### 4.3 Web 전용 고려사항

- **반응형 UI**
  - 모바일/태블릿/데스크톱 화면폭에 맞는 레이아웃 필요
- **브라우저 저장소**
  - 학생 정보, 캐시, 타이머 상태를 웹 환경에 맞는 저장 방식으로 관리해야 함
  - 예: `localStorage`, `sessionStorage`, `IndexedDB` 등 검토
- **캐시/복구 UX**
  - 데이터가 캐시에서 왔는지, 마지막 성공 데이터를 대신 보여주는지 사용자에게 읽기 쉬운 문구로 알려야 함
- **웹 알림**
  - 타이머 종료 알림이 필요하면 브라우저 Notification API 또는 대체 UX 검토
- **위젯 대체 UX**
  - Android AppWidget은 웹에 그대로 옮길 수 없으므로, 웹에서는 다음 중 하나로 대체 검토
    - 홈 대시보드 고정 카드
    - PWA 홈 화면 추가
    - 빠른 접근용 요약 위젯성 영역

### 4.4 Web 개발 기준

- 저장소 루트에 `web/` 폴더를 두고 웹 전용 구현을 시작합니다.
- Android의 도메인 개념은 최대한 유지하되, UI/저장/알림 방식은 웹 플랫폼 제약에 맞게 분리합니다.
- 외부 NEIS API는 브라우저에서 직접 호출하지 않고 `app/api/**/route.ts` 경유 구조를 유지합니다.

## 5. 공통 개선 과제 및 플랫폼별 확장 포인트

### 5.1 공통 개선 과제

- **학교 코드 하드코딩 해제**
  - 현재 특정 교육청/학교 코드가 고정되어 있으므로, 여러 학교를 지원하려면 학교 검색 및 동적 저장 구조가 필요합니다.
- **에러 핸들링 강화**
  - API 키 제한, 네트워크 오류, 응답 없음 등 상황별 안내를 더 정교하게 구성할 필요가 있습니다.
- **나이스 응답 코드 해석 보강**
  - 단순 메시지 노출을 넘어서 재시도 유도, 설정 변경 유도 등 액션 가능한 안내로 확장 필요
- **다크모드/테마 지원 강화**
  - 플랫폼별 테마 시스템에 맞춰 일관된 다크모드 경험 필요

### 5.2 Android 확장 포인트

- 위젯 옵션 확대
- 화면 크기별 레이아웃 분기 고도화
- 알림/백그라운드 동작 세밀화

### 5.3 Web 확장 포인트

- 반응형 네비게이션 구조 설계
- PWA 적용 여부 검토
- 브라우저 저장소/캐시 전략 최적화
- 모바일 웹에서의 빠른 진입 UX 개선
- 공통 도메인 기준을 Android와 다시 맞추기
