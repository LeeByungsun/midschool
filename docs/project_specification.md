# MisSchoolApp 프로젝트 분석 및 기획서

현재 구현된 프로젝트 `MisSchoolApp`의 소스 코드를 분석한 결과를 바탕으로 작성된 기획서(Project Specification) 및 향후 개발 플랜입니다. 

## 1. 프로젝트 개요 (Overview)
- **앱 이름**: 중학교도우미
- **주요 타겟**: 특정 중학교 재학생 (현재 교육청 및 학교 코드가 하드코딩되어 있어 단일 학교 타겟으로 작동)
- **서비스 목표**: 중학생들이 매일 확인해야 하는 시간표, 급식, 학사 일정 정보 조회 기능과, 자기주도학습을 돕는 스터디 타이머를 하나의 앱에서 편하게 제공하는 것을 목표로 합니다.

## 2. 주요 기능 명세 (Features)

### 2.1 온보딩 및 앱 설정
- **스플래시 화면 (`SplashActivity`)**: 앱 진입 시 초기 데이터 세팅 및 권한 체크용
- **초기 설정 (`SetupActivity`)**: 처음 앱 실행 시 학년과 반 정보를 입력받아 저장소 계층(`PreferencesRepository`)을 통해 로컬에 저장합니다.
- **설정 (`SettingsActivity`)**: 학교 정보, 학년/반 수정 및 타이머 동작 관련(소리, 진동 등) 상세 설정을 관리합니다.

### 2.2 메인 대시보드 (`MainActivity`)
- 대시보드 형식으로 주요 정보를 한눈에 위젯처럼 요약 제공:
  - 오늘 날짜, 오늘 수업(시간표) 요약
  - 오늘 급식 메뉴 및 영양/칼로리 정보 요약
  - 다가오는 학사 일정(이벤트) 요약 제공
  - 단축 버튼들을 통해 `TimetableActivity`, `ScheduleActivity` 등 상세 화면으로 즉시 이동 가능

### 2.3 스터디 타이머 (`timer/` & `MainActivity`)
- **모드 프리셋**: 집중(Focus), 휴식(Break), 딥포커스(Deep Focus) 프리셋 지원
- **상태 표시**: 숫자형 타이머 뷰와 링형 커스텀 뷰(`TimerRingView`) 교차 지원
- **알림 기능**: `TimerAlarmReceiver`를 통한 백그라운드 타이머 종료 알림(로컬 노티피케이션) 및 진동 지원
- **상태 유지**: 앱을 닫거나 다시 열어도 진행 중이던 타이머 상태를 복원합니다.

### 2.4 학교 정보 상세 내역
- **시간표 (`TimetableActivity`)**: 사용자의 학년/반 정보를 기준으로 나이스 API를 통해 날짜 이동형 일간 시간표를 조회합니다.
- **학사 일정 (`ScheduleActivity`)**: 이번 달 학교의 주요 행사 및 일정 목록 제공
- **로컬 캐시**: 급식과 시간표는 최근 조회 결과를 로컬에 캐시해 네트워크 오류 시 마지막 성공 데이터를 우선 활용합니다.
- **캐시 만료 정책**: 급식 캐시는 12시간, 시간표 캐시는 24시간 동안만 재사용하며, 만료된 데이터는 자동으로 폐기합니다.

### 2.5 바탕화면 위젯 (`widget/` 패키지)
- **홈 화면 위젯 (AppWidget)**: 앱을 열지 않고도 스마트폰 바탕화면에서 오늘과 내일 시간표를 구분해서 한눈에 볼 수 있는 정보성 위젯 제공
- **새로고침**: 위젯 내 새로고침 버튼을 터치하면 백그라운드 코루틴 환경에서 실시간으로 나이스 API를 재호출해 최신 데이터를 갱신합니다.
- **오류 상태 처리**: 학년/반 미설정, 시간표/급식 데이터 없음, 네트워크/나이스 응답 오류를 구분해 위젯에 사용자용 문구로 노출합니다.
- **위젯 설정 화면**: 위젯 추가 시 `내일 시간표 표시` 옵션을 선택할 수 있습니다.
- **위젯 재설정**: 이미 추가된 위젯에서도 상단 설정 아이콘을 눌러 같은 설정 화면을 다시 열 수 있습니다.
- **작은 위젯 대응**: 급식 영역을 제거하고 위젯 높이를 늘려 오늘/내일 시간표 전체를 더 안정적으로 보여주도록 구성합니다.
- **과목명 길이 제한**: 위젯에서는 과목명이 6자를 넘으면 5글자까지만 잘라 표시해 줄바꿈과 잘림을 줄입니다.

## 3. 시스템 아키텍처 및 기술 스택
- **언어 및 환경**: Kotlin, Android SDK (minSdk 26, targetSdk 36)
- **디자인 패턴 / 아키텍처**: MVVM (Model-View-ViewModel) 패턴을 충실히 따르며 데이터와 UI 계층을 분리
- **UI 바인딩**: 주요 화면은 DataBinding과 ViewModel 조합으로 정리해 Activity는 이벤트 연결과 화면 전환 책임 위주로 유지합니다.
- **저장소 계층**: 학교 데이터는 `SchoolRepository`, 로컬 설정/캐시는 `PreferencesRepository`를 통해 접근합니다.
- **비동기 처리**: `runCatching`, Kotlin Coroutines (`lifecycleScope`), Flow (`StateFlow` 등)
- **네트워크 / 데이터 연동**: 
  - 서드파티 라이브러리: Retrofit2 + OkHttp3 + Gson
  - 데이터 소스: [나이스(NEIS) 교육정보 개방 포털 API](https://open.neis.go.kr/)
  - 연동 API: 급식(`hub/mealServiceDietInfo`), 학사일정(`hub/SchoolSchedule`), 중학교 시간표(`hub/misTimetable`)
- **의존성 주입**: Dagger Hilt (`NetworkModule` 등 활용)
- **로컬 스토리지**: `SharedPreferences` 기반 저장을 `PreferencesRepository` 계층으로 감싼 구조로 운영하며, 사용자 설정/위젯 설정/간단 캐시에 활용합니다.
- **캐시 전략**: 날짜 기준 응답을 그대로 보관하되 TTL(Time To Live)을 두어 오래된 급식/시간표 데이터가 계속 노출되지 않도록 제어합니다.

## 4. 확인된 취약점 및 개선 방안 (Future Plans)
- **학교 코드 하드코딩 해제**: 현재 `NetworkModule`에 "J10" (교육청), "7679399" (학교) 코드가 고정으로 주입되어 있습니다. 여러 학교에서 사용할 수 있도록 론칭하려면, 온보딩 단계에 학교 이름 검색 API를 연동하여 동적으로 학교 정보를 저장하도록 아키텍처 변경이 필요합니다.
- **에러 핸들링 및 상태 처리 강화**: API 통신 에러 발생(키 제한, 네트워크 오류 등) 시 사용자에게 스낵바 또는 명시적인 에러 UI를 띄우는 처리가 추가 고도화되어야 합니다.
- **나이스 응답 코드 파싱 보강**: 현재는 공통 저장소에서 NEIS `RESULT.CODE`를 해석해 사용자 친화적인 예외 메시지로 변환하지만, 코드별 세분화된 대응(예: 재시도/인증키 재설정 유도)은 추가 개선 여지가 있습니다.
- **다크모드 맞춤 태마 지원**: 추가적인 시스템 테마 연동을 통해 라이트모드와 다크모드 환경에서 유기적으로 색상이 변경되도록 지원할 계획입니다.
- **위젯 옵션 추가 확장**: 현재는 정보 노출 여부와 표시 과목 수만 제어하므로, 추후 오늘 전용 레이아웃이나 위젯 크기별 레이아웃 분기, 수동/자동 갱신 정책도 확장할 수 있습니다.
